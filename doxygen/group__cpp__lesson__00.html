Lesson 00 – Configuración del entorno de trabajo y flujo Fork + Upstream

Objetivos

Al finalizar esta lección, el estudiante será capaz de:

  • Instalar y verificar las herramientas necesarias para el curso
  • Crear un fork del repositorio oficial del curso
  • Clonar su fork y configurar el remoto upstream
  • Sincronizar actualizaciones del curso desde upstream
  • Compilar el proyecto localmente usando CMake (build fuera del árbol)
  • Ejecutar pruebas localmente usando CTest
  • Comprender y respetar la regla de protección del curso
  • Realizar su commit inicial correctamente

Requisitos mínimos

Se requiere:

  • Acceso a una terminal tipo UNIX (Linux/macOS o Windows con WSL2)
  • Un compilador de C++ (GCC o Clang)
  • Git
  • CMake

Recomendado:

  • Un editor de código (VS Code, CLion, etc.)
  • Un depurador (gdb o lldb)

Sistemas operativos soportados

El curso está diseñado para funcionar en:

  • Linux (recomendado)
  • macOS
  • Windows usando WSL2 (recomendado)

Instalación de herramientas

Linux (Ubuntu/Debian)

sudo apt update
sudo apt install -y build-essential git cmake gdb

macOS

xcode-select --install
brew install git cmake

Windows (WSL2 + Ubuntu)

sudo apt update
sudo apt install -y build-essential git cmake gdb

Nota:

  • Trabaja dentro de /home/<usuario>/ para mejor rendimiento.

Verificación del entorno

Verifica que las herramientas estén correctamente instaladas:

g++ --version
git --version
cmake --version

Si alguno de estos comandos falla, el entorno NO está listo.

Modelo del curso: Fork + Upstream

Este curso utiliza un modelo profesional de trabajo con GitHub:

  • Existe un repositorio oficial del curso (upstream)
  • Cada estudiante trabaja en un fork personal
  • El estudiante puede recibir actualizaciones del curso

Repositorio oficial (upstream):

  • https://github.com/ChecheRomo96/cpp-foundations

Este repositorio contiene:

  • CI (GitHub Actions)
  • CMake y configuración base
  • Documentación oficial del curso
  • Reglas y protecciones del curso

Crear el fork (GitHub)

  1. Entra al repositorio oficial del curso: https://github.com/ChecheRomo96/cpp-foundations
  2. Haz clic en Fork
  3. Crea el fork bajo tu cuenta de GitHub

Tu fork será tu repositorio de trabajo:

  • https://github.com/<tu-usuario>/cpp-foundations

Clonar tu fork (trabajo local)

git clone https://github.com/<tu-usuario>/cpp-foundations.git
cd cpp-foundations

Verifica los remotos:

git remote -v

Debes ver:

  • origin -> tu fork

Configurar upstream

Agrega el repositorio oficial como upstream:

git remote add upstream https://github.com/ChecheRomo96/cpp-foundations.git
git remote -v

Debes ver:

  • origin -> tu fork
  • upstream -> repositorio oficial

Actualizar desde la terminal

Cuando el instructor publique cambios:

git fetch upstream
git merge upstream/main

Nota:

  • El estudiante solo debe modificar student/
  • Los conflictos deberían ser raros si se respeta esta regla

Actualizar desde GitHub (Sync fork)

GitHub ofrece una forma visual de sincronizar el fork:

  1. Entra a tu fork: https://github.com/<tu-usuario>/cpp-foundations
  2. Haz clic en Sync fork
  3. Selecciona Update branch

Después, actualiza tu copia local:

git pull origin main

Este método es útil si aún no dominas Git por terminal.

El archivo .gitignore

Durante el proceso de compilación, el proyecto genera archivos temporales y específicos del entorno local que no forman parte del código fuente y no deben subirse al repositorio.

El archivo **.gitignore** le indica a Git qué archivos y carpetas deben ser ignorados, es decir, no rastreados ni versionados, incluso si existen en tu sistema local.

El uso correcto de .gitignore es fundamental para:

  • Mantener el repositorio limpio y legible
  • Evitar subir archivos dependientes de tu sistema
  • Prevenir conflictos entre distintos entornos de desarrollo
  • Reducir ruido innecesario en el historial de commits

Archivos de compilación

Durante el proceso de compilación, el proyecto genera archivos automáticos y dependientes del entorno local.

Estos archivos no forman parte del código fuente y no deben versionarse bajo ninguna circunstancia.

En este curso, los archivos generados por compilación deben permanecer exclusivamente en el entorno local del estudiante.

En particular, la carpeta:

build/

nunca debe subirse al repositorio.

Compilación local con CMake

Flujo recomendado (build fuera del árbol):

mkdir -p build
cmake -S . -B build
cmake --build build

Ejecución de pruebas locales

Ejecutar pruebas:

ctest --test-dir build --output-on-failure

Arreglar un CI fallido después de un merge

Es común que el CI falle cuando, después de hacer un merge o pull, se hayan incluido cambios fuera de la carpeta student/.

Esto suele ocurrir por:

  • Usar git add .
  • Hacer un merge desde GitHub que arrastra archivos generados
  • Resolver conflictos sin cuidado

La siguiente secuencia de comandos permite corregir el problema sin perder el trabajo del alumno.

Secuencia de corrección estándar

Ejecuta estos comandos desde la raíz del repositorio:

git restore --staged .
git restore .
git add student/
git commit -m "Fix: only student changes"
git push

Explicación:

  • git restore –staged . : quita todos los archivos del staging
  • git restore . : revierte archivos modificados al estado del repositorio
  • git add student/ : agrega únicamente el trabajo del alumno
  • git commit : crea un commit limpio y válido
  • git push : sube los cambios corregidos

Si después de esto el CI sigue fallando, revisa el mensaje del workflow para identificar archivos adicionales fuera de student/.

Regla de oro

En este curso:

  • SIEMPRE usar:
git add student/
  • NUNCA usar:
git add .

El CI está diseñado para fallar si esta regla no se cumple.
